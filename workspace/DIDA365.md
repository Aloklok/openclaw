# DIDA365.md - 滴答清单操作规范

## 1. 🔧 Skill 管理规则
- **删除Skill前必须确认** - 当用户说"关闭Skill"时，指的是暂时关闭而非删除
- **安装Skill使用Find Skill** - 优先使用Find Skill工具搜索，而不是直接到ClawHub查找
- **Skill操作需谨慎** - 任何Skill的删除操作都需要先和用户确认

## 2. 📅 任务时间调整规范
- **时间统一原则** - 调整任务时间时，必须同时更新 startDate 和 dueDate
- **任务前移** - 将任务提前时，整个任务的时间段都要相应前移
- **任务后移** - 将任务推后时，整个任务的时间段都要相应后移
- **完整性检查** - 修改后必须验证两个日期都已正确更新

## 3. 📅 滴答清单集成规则

#### 工具分工准则
- **dida365 CLI** - 专门用于标签相关操作（标签管理、父子标签、标签体系维护）
- **dida365-openai** - 默认用于所有其他任务操作（创建、更新、移动、日期调整、备注等）

#### 时间格式规范
- **dida365-openai格式**：`YYYY-MM-DDTHH:mm:ss+0000`（例如：`2026-04-01T08:00:00+0000`）
- **dida365 CLI格式**：`YYYY-MM-DDTHH:mm:ss.000+0000`（例如：`2026-04-01T15:59:59.000+0000`）

#### 任务备注处理
- **desc字段**：任务的简短描述，在桌面端显示位置小，适合简短说明（如执行日期）
- **content字段**：任务的主要内容或详细说明，记录过程信息
- **item字段**：子任务列表，用于将复杂任务分解为多个步骤，每个子任务可单独标记完成
- **更新方法**：desc字段推荐使用 `--json '{"desc": "说明内容"}'` 格式，更稳定
- **格式建议**：desc简短精炼，content可包含详细信息

#### 自动触发
- **任务添加自动调用** - 当用户提到"添加任务"、"准备去干的事情"或"什么时候要去干什么"时，自动调用滴答清单工具
- **保持认证状态** - 确保滴答清单工具始终保持认证状态，避免重复认证

## 4. 🏷️ 标签体系（知识管理体系）

**标签创建规则：**
- 每次新建任务时，评估其所属标签类别
- 如果没有合适的现有标签，向用户提出新建标签建议
- 随着知识点增多，定期评估标签的合并、拆分或优化需求
- 标签支持父子层级结构，使用 `--parent` 参数创建

**标签操作原则：**
- ✅ 新建标签：用户确认后使用 `npx dida365 tag create "标签名" --parent "父标签"`
- ❌ 删除标签：必须由用户确认，使用 `npx dida365 tag delete "标签名"`
- ❌ 禁止空标签：严禁创建没有任务的空子标签
- ✅ 优先复用：新建任务时优先使用现有标签
- 🔄 修改标签：向用户提出优化建议，确认后执行

## 5. 📅 任务查询最佳实践

**高效查询方法：**
- 先使用 `npx dida365 sync all` 获取所有任务数据
- 直接筛选 due date 为特定日期的任务
- 避免逐个查看项目再查找的低效方式
- 对于今日任务，优先查看收件箱和即将到期的任务

## 6. 📁 清单体系（大类归纳）
- 清单用于任务的大类归纳和容器管理
- 每个清单对应一个主要的工作或学习领域
- 新建任务时，根据任务性质推荐合适的清单
- 删除清单必须由用户确认

## 7. 🏷️ 标签分类体系（学习与知识管理）
- `🏗️ 系统设计` - DDD、架构模式、系统建模等
- `☁️ 云原生` - 云基础设施、分布式系统、FinOps等
- `📊 数据智能` - 数据治理、数据库、推荐系统等
- `🛠️ 工程技术` - DevOps、工具链、CI/CD等
- `📋 项目管理` - 流程优化、敏捷管理、工具使用等
- `🏭 产业应用` - 智慧城市、政务数字化、产业互联网等
- `🚀 前仰技术` - 物联网、元宇宙、区块链、在线学习等

## 8. 🎯 总体流程建议
- **判断任务性质** - 技术任务 (推荐1-3个标签 + 1个清单) / 临时事务 (放入收集箱)
- **子任务关系** - 使用 `npx dida365 batch set-subtask` 建立父子关系，创建后务必执行 `sync all`。
- **重新安排** - 更新时间时必须同时设置 `startDate` 和 `dueDate` 以保持协调。
